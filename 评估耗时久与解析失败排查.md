# 评估耗时久与解析失败排查（Interview Evaluate）

本文记录“面试评估很久/一直不出结果”的排查结论与修复建议，便于后续复盘与稳定化。

## 1. 现象

- 前端显示评估耗时很久，或迟迟不出报告。
- 后端日志中 `evaluate-consumer` 长时间无“评估完成/保存报告”的成功日志。

## 2. 直接原因（从日志可见）

评估任务并非卡死，而是：

- 调用大模型生成 **很长的结构化 JSON**（题目多、字段多、文本长），导致生成时间变长；
- 且出现 **结构化解析失败（JSON 不完整/被截断）** → 触发重试，进一步拉长耗时。

### 2.1 日志证据（示例）

评估任务开始：

```text
开始处理评估任务: sessionId=..., messageId=..., retryCount=0
开始评估面试: <sessionId>, 共16题
```

模型返回后解析失败（关键在 `BeanOutputConverter`）：

```text
Could not parse the given text to the desired target type: "{ ... }"
JsonMappingException: Unexpected end-of-input within/between Object entries
批次评估结构化解析失败，准备重试: attempt=1, error=...
```

> `Unexpected end-of-input` 基本等价于：返回内容在 JSON 结构上“没闭合/被截断/不完整”。

## 3. 为什么会这样（常见根因）

### 3.1 输出过大导致截断

评估 DTO 往往包含：

- `questionEvaluations[]`（题数越多越大）
- 每题的 `feedback`、`keyPoints[]`、`referenceAnswer`（尤其 `referenceAnswer` 很长）
- `overallFeedback`、`improvements[]` 等总结内容

当一次性让模型输出“16 题完整评估 + 参考答案 + 要点 + 总结”，很容易超过模型/网关的输出上限或触发截断。

### 3.2 结构化约束不够强 / 格式漂移

即便提供了格式说明，模型仍可能输出：

- 多余文本（解释/前后缀）
- Markdown 代码块
- JSON 中出现未转义字符
- 最常见：输出中途被截断导致 JSON 不闭合

### 3.3 超时/网络中断导致响应不完整

如果 HTTP 层超时或连接被中断，也可能产生“半截 JSON”。

## 4. 最优修复方案（按收益/稳定性排序）

### 方案 A（推荐）：拆批评估 + 最终汇总

将 16 题拆成多个批次（例如每批 4 题）分别生成结构化评估，再做一次“汇总”：

- **好处**
  - 单次输出显著变短，截断概率大幅下降
  - 失败只重试该批次，成本更低
  - 总耗时更稳定
- **代价**
  - 需要改服务层逻辑（批次循环、聚合 DTO）

### 方案 B：减少/裁剪返回字段（尤其 `referenceAnswer`）

将 `referenceAnswer` 改为：

- 只返回关键点（bullet）
- 或限制长度（例如每题最多 200~400 字）

> 这通常能立刻降低输出体积，让解析更稳定。

### 方案 C：更严格的“仅 JSON”约束与重试提示

在 system prompt 中明确：

- 仅返回 JSON（禁止 markdown、禁止解释）
- 字段必须完整、禁止省略
- 强调转义规则

并在重试 prompt 中附带上次失败原因（例如“JSON 不完整/Unexpected end-of-input”），要求模型重新输出完整 JSON。

### 方案 D：限制模型输出上限/超时参数（若可配）

在 Spring AI/OpenAI options 中设置合理的：

- max tokens（避免默认上限导致截断）
- timeout（避免 HTTP 层提前中断）

## 5. 推荐落地路径（最小改动优先）

1) **先做方案 B（裁剪字段/长度）**：最快见效，改动面最小  
2) 若仍不稳定，再做 **方案 A（拆批）**：稳定性最强  
3) 同时加强 **方案 C（严格 JSON）** 作为兜底  

## 6. 如何验证修复有效

- 触发一次评估任务后，观察日志：
  - 不再出现 `BeanOutputConverter Could not parse...`
  - 评估完成日志在可预期时间内出现
- 前端侧：
  - 报告页能稳定加载（不需要多次刷新/等待极久）
