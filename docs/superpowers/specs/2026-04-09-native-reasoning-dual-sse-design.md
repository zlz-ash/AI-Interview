# 原生推理 + 双通道 SSE 设计说明

## 1. 背景与目标

部分上游大模型或网关在流式输出中会**单独给出推理/思考内容**（例如 OpenAI 兼容接口里 `delta` 中的 `reasoning`、`reasoning_content` 等字段），与面向用户的**正文**并行增量下发。

本设计目标：

- **后端**不再只消费 Spring AI `ChatClient` 流上的 `.content()` 单通道，而是在可行层级**消费完整流式响应**，将「推理」与「正文」拆成统一内部片段，再通过 **SSE** 下发。
- **前端**分区展示：上方可折叠「思考」区随推理增量更新，下方 Markdown 正文区随正文增量更新。
- **换模型**时对内固定自有协议，对外用**适配层**映射各厂商字段；无推理模型时推理为空，界面自动退化为仅正文。
- **持久化策略（已确认）**：推理**不落库**、历史接口**不返回**推理全文；仅将**正文全文**写入现有消息内容字段（与当前行为一致）。

## 2. 已定决策

| 项目 | 决策 |
|------|------|
| SSE 载荷形态 | **方案二**：每条 SSE 仅使用 `data:` 行，载荷为 **JSON 字符串**（信封结构见下），不依赖 `event:` 行区分类型（降低 fetch 流解析状态机复杂度）。 |
| 推理持久化 | **A**：不持久化；流结束后思考内容仅存在于当次会话前端状态，刷新或重进会话后只依赖已存正文。 |

## 3. 对内流式协议（固定）

### 3.1 片段类型

- `reasoning`：可选。表示模型内部推理/思考文本的**增量**。
- `content`：主答案通道。表示对用户展示的正文**增量**（通常为 Markdown）。

语义约定：

- 二者均可出现零长度增量；**无推理模型**可不发送 `reasoning` 或仅发送空增量，前端不展示思考区。
- **完成流式时**，以后端写入库的字符串为准：**仅拼接 `content` 增量**得到完整正文并调用现有 `completeStreamMessage`（或等价逻辑）。

### 3.2 SSE `data:` JSON 信封（每帧）

建议字段（可扩展，新增字段须向后兼容）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | `reasoning` \| `content` |
| `delta` | string | 本帧 UTF-8 文本增量（允许空字符串） |

示例：

```json
{"type":"reasoning","delta":"首先"}
{"type":"content","delta":"根据"}
```

### 3.3 流结束

- 保持与现网兼容：在正文流结束后发送**最后一帧**约定结束标记（例如现有 `[DONE]` 约定），或发送带 `type` 的结束帧；**须在实现计划中固定一种**，前后端统一解析。
- 结束帧**不参与**正文拼接。

## 4. 后端架构

### 4.1 适配层（对外）

- **职责**：将各厂商/网关的原始流式 chunk（JSON 行或框架对象）映射为统一内部表示：`StreamPart { type: reasoning|content, delta: string }`。
- **无推理**：不产出 `reasoning` 或产出空 `delta` 的 `reasoning` 帧（实现任选其一，前端行为一致）。

### 4.2 模型消费（对内）

- **必须**能访问「完整 delta」，而不仅是聚合后的 `content()` 字符串。若当前 `ChatClient` / `ChatModel` 未暴露推理字段，则采用以下之一（实现阶段选型）：
  - 使用支持该字段的 **Spring AI 扩展**或**自定义 `ChatModel`**；或
  - 在合规前提下对 **OpenAI 兼容 HTTP SSE** 做直连解析，再接入适配层。
- 本设计不绑定具体厂商字段名；适配层集中维护映射表。

### 4.3 SSE 发射

- 将 `Flux<StreamPart>`（或等效）映射为 `Flux<ServerSentEvent<String>>`，每帧 `data` 为**单行 JSON 字符串**（信封），并对 JSON 内特殊字符正确转义。
- 流式过程中内存中可用两个 `StringBuilder` 分别累积 reasoning 与 content（**仅用于调试或可选日志，不落库**）；**持久化仅写入 content 的完整拼接结果**。

### 4.4 错误处理

- 若某一帧 JSON 解析失败：记录日志；策略二选一（实现时固定）：**跳过该帧**或**终止整流并返回可读错误**（须在实现计划中写明）。
- 若长时间仅有 `reasoning` 而无 `content`：前端正文区可显示加载态；**超时**策略属实现细节，本设计不强制秒数。

## 5. 前端表现

### 5.1 布局与行为

- **上方**：可折叠区域，标题如「思考」；订阅 `type === "reasoning"` 的 `delta` 增量追加显示（纯文本或轻量格式化按产品定，默认纯文本）。
- **下方**：Markdown 渲染区；订阅 `type === "content"` 的增量。
- **无推理**：从未收到非空 `reasoning` 时**不渲染**思考区（避免占位闪烁）。

### 5.2 解析

- 延续 **fetch + ReadableStream**；按 SSE 事件块解析，从 `data:` 行取出字符串后 **`JSON.parse`** 得到信封，按 `type` 分发到两个本地 state。
- 与结束标记的衔接见 §3.3。

## 6. 范围建议（实现计划可裁剪）

- **首期**：与当前产品一致，优先覆盖 **RAG 对话流式**（`/api/rag-chat/.../messages/stream`）及 **知识库 query stream**（若共用同一套查询服务，则一并改为双通道协议）。
- **其他 LLM 流**（如面试、简历分析）：非本设计强制范围；若复用同一 SSE 封装，可后续按需接入。

## 7. 测试要点

- **适配层单元测试**：模拟多种上游 delta（仅 reasoning、仅 content、交错、空增量）→ 统一 `StreamPart` 序列。
- **集成测试**：Mock 上游 SSE，断言 SSE 出站帧序列与 DB 中仅存正文。
- **前端**：无 `reasoning` 时不出现思考区；有 `reasoning` 时折叠/展开与增量更新正确。

## 8. 非目标（本期）

- 推理内容检索、导出、审计长期存储。
- 跨会话「思考」回放。
- 强制各模型输出推理（依赖上游能力）。
