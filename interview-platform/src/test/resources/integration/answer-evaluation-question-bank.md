# Answer Evaluation 集成测试题目集（草案 v1）

说明：

- 题号使用 0-based，便于直接映射 `InterviewQuestionDTO.questionIndex`。
- 每题都包含“标准化用户回答”，用于稳定复现评估流程。
- 该题集分为 `smoke-6`（快速验证）和 `full-16`（接近真实压力）。

## smoke-6（快速集成验证）

### Q0（PROJECT）

- 分类：项目经历
- 问题：请介绍你在最近一个后端项目中的核心职责，以及你最有价值的技术决策。
- 用户回答：我在一个招聘面试平台项目中主要负责后端架构设计、AI 评估链路和数据模型定义。最有价值的决策是把“答案评估”从单次大请求改成分批评估再汇总。这样做后，结构化 JSON 的单次输出长度明显下降，解析失败率降低。我们还把失败重试限制在单批次范围内，避免整份报告重算，整体耗时更稳定。

### Q1（JAVA_BASIC）

- 分类：Java基础
- 问题：`equals` 和 `hashCode` 的契约是什么？如果违反会导致什么问题？
- 用户回答：契约核心是：相等对象必须有相同 hashCode；同一次运行中 hashCode 尽量稳定；hashCode 相同对象不一定相等。违反契约会导致 HashMap、HashSet 行为异常，比如 put 了对象却 get 不到，或者集合中出现“逻辑重复但去重失败”的情况。

### Q2（JAVA_COLLECTION）

- 分类：集合
- 问题：`ArrayList` 和 `LinkedList` 在插入、随机访问场景下怎么选？
- 用户回答：随机访问高频优先 ArrayList，因为连续内存+索引访问快。中间插入删除高频且迭代器定位后操作多，LinkedList 才可能有优势。但在现代 CPU 缓存下，很多业务场景 ArrayList 实测更好，所以通常默认 ArrayList，只有明确链表收益才选 LinkedList。

### Q3（JAVA_CONCURRENT）

- 分类：并发
- 问题：线程池参数如何配置？你会重点关注哪些指标？
- 用户回答：我一般先从任务类型区分 CPU 密集或 IO 密集。CPU 密集接近核心数，IO 密集可以更大。重点看 activeCount、队列长度、任务等待时间、拒绝次数和 P95 延迟。拒绝策略倾向先保护系统，比如 CallerRuns 或降级兜底，同时给出监控告警。

### Q4（MYSQL）

- 分类：MySQL
- 问题：说说你做过的一次 SQL 优化，具体怎么定位和验证效果？
- 用户回答：我遇到过分页查询慢的问题，先用 explain 看执行计划，发现走了回表+filesort。后面通过联合索引覆盖查询列，并把深分页改成基于游标的“上次最大 id”翻页。上线前后对比慢 SQL 日志和接口 RT，P95 从 800ms 降到 120ms 左右。

### Q5（SPRING_BOOT）

- 分类：Spring Boot
- 问题：在 Spring Boot 项目里你如何处理配置管理和环境隔离？
- 用户回答：我会用 application.yml + profile 分层，默认只保留公共配置。敏感信息全部走环境变量或密钥管理，不写死到仓库。再配合 `@ConfigurationProperties` 做结构化配置，避免散落 `@Value`。本地、测试、生产分别启用 profile，确保连接串和开关不串环境。

## full-16（真实链路验证）

### Q0（PROJECT）

- 分类：项目经历
- 问题：你做过最有挑战的需求是什么？最后如何落地？
- 用户回答：最有挑战的是 AI 面试评估报告稳定性。初版一次性生成 16 题结构化 JSON，经常因输出过长出现截断，触发解析失败和重试，导致用户等待很久。我们改为“分批评估 + 汇总”，并增加严格 JSON 约束与失败降级。最终失败率明显下降，整体体验稳定。

### Q1（PROJECT）

- 分类：项目经历
- 问题：你如何和前端、产品协作定义“评估可用”标准？
- 用户回答：我们把“可用”拆成可衡量指标：成功率、耗时、错误可解释性。前端希望 95% 请求在可接受时间内返回，产品希望失败时也有兜底提示。于是后端增加了题目级降级策略、日志埋点和批次耗时统计，前端按状态展示“处理中/完成/部分降级”。

### Q2（JAVA_BASIC）

- 分类：Java基础
- 问题：`StringBuilder`、`StringBuffer`、`String` 的差异和使用建议是什么？
- 用户回答：String 不可变，适合常量和读多写少。StringBuilder 非线程安全但性能好，适合单线程拼接。StringBuffer 线程安全有锁，通常只在多线程共享拼接场景才考虑。业务代码里高频拼接我基本用 StringBuilder。

### Q3（JAVA_BASIC）

- 分类：Java基础
- 问题：`try-with-resources` 的优势是什么？底层行为你了解吗？
- 用户回答：优势是自动关闭资源，减少泄漏。底层会在编译后生成 finally 关闭逻辑，并处理 suppressed exceptions，保证主异常不被覆盖。处理 IO、数据库连接、流对象时非常实用。

### Q4（JAVA_COLLECTION）

- 分类：集合
- 问题：`HashMap` 在 JDK8 的结构和扩容机制是什么？
- 用户回答：JDK8 是数组+链表+红黑树，冲突链过长会树化。扩容时容量翻倍，元素根据高位 bit 重新分布到原位置或原位置+oldCap，减少重算成本。初始容量建议按预估数据量设置，避免频繁扩容。

### Q5（JAVA_COLLECTION）

- 分类：集合
- 问题：`ConcurrentHashMap` 为什么比 `Hashtable` 更适合高并发？
- 用户回答：Hashtable 基本是全表锁，竞争激烈。ConcurrentHashMap 在 JDK8 里使用 CAS + synchronized 局部控制，冲突热点才加锁，整体并发度更高。读操作多数无锁，吞吐明显更好。

### Q6（JAVA_CONCURRENT）

- 分类：并发
- 问题：你如何理解 volatile 的可见性和有序性？
- 用户回答：volatile 保证写后对其他线程可见，并通过内存屏障限制重排序，适合状态标记和双检锁中的引用发布。但它不保证复合操作原子性，比如 i++ 仍需原子类或锁。

### Q7（JAVA_CONCURRENT）

- 分类：并发
- 问题：synchronized 和 ReentrantLock 的取舍？
- 用户回答：synchronized 语义简单、JVM 优化成熟，很多场景够用。ReentrantLock 提供可中断、公平锁、尝试加锁和条件队列，适合需要更细控制的并发流程。一般先用 synchronized，遇到明确需求再换 Lock。

### Q8（MYSQL）

- 分类：MySQL
- 问题：事务隔离级别及其典型问题有哪些？
- 用户回答：读未提交会脏读；读已提交解决脏读但有不可重复读；可重复读在 MySQL 下配合 MVCC 能避免大部分不可重复读；串行化最安全但并发最低。线上通常用可重复读，再结合业务锁和唯一约束避免写冲突。

### Q9（MYSQL）

- 分类：MySQL
- 问题：什么情况下会出现索引失效？
- 用户回答：常见有函数操作索引列、前导模糊 like、隐式类型转换、范围条件后再用后续列、OR 条件未命中复合索引等。排查时我会结合执行计划、慢查询日志和实际参数看是否命中预期索引。

### Q10（REDIS）

- 分类：Redis
- 问题：缓存穿透、击穿、雪崩分别怎么处理？
- 用户回答：穿透用布隆过滤器或空值缓存；击穿对热点 key 加互斥重建或逻辑过期；雪崩通过过期时间打散、多级缓存和限流降级。关键是要把“数据库保护”放在第一位，避免缓存故障拖垮核心库。

### Q11（REDIS）

- 分类：Redis
- 问题：你会怎么设计一个分布式锁？
- 用户回答：核心是 set nx px + 唯一 value，释放锁时校验 value 避免误删。要考虑锁续约、超时和业务幂等，必要时引入 Redisson。对一致性要求极高的场景，还需要评估锁方案是否满足故障语义。

### Q12（SPRING）

- 分类：Spring
- 问题：Spring AOP 的实现原理和典型应用场景？
- 用户回答：Spring AOP 基于代理模式，接口走 JDK 动态代理，类走 CGLIB。典型场景是日志、鉴权、限流、事务。我们项目里就用 AOP 做限流和链路埋点，避免重复横切代码。

### Q13（SPRING）

- 分类：Spring
- 问题：事务传播行为里 `REQUIRES_NEW` 和 `REQUIRED` 的区别？
- 用户回答：REQUIRED 复用当前事务，没有就新建；REQUIRES_NEW 总是挂起外层并新开事务。前者适合业务整体一致性，后者适合独立提交场景（比如审计日志），但要注意连接占用和嵌套复杂度。

### Q14（SPRING_BOOT）

- 分类：Spring Boot
- 问题：你会如何做健康检查和可观测性？
- 用户回答：至少接入 actuator 暴露健康、指标和自定义探针。再通过日志规范（traceId）+ 指标（成功率、耗时、错误码）+ 告警阈值构建闭环。关键链路要有分段耗时，便于定位是模型慢还是解析慢。

### Q15（SPRING_BOOT）

- 分类：Spring Boot
- 问题：请讲一下你对配置热更新和灰度发布的实践理解。
- 用户回答：配置热更新要先区分“可热更”和“需重启”，避免误改关键连接参数。灰度发布建议按流量比例或用户分组逐步放量，观察错误率和延迟后再全量。对于 AI 评估链路，可以先灰度新提示词和批次策略，确保稳定后再推广。
